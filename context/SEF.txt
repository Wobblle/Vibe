PURPOSE
--------------------------------------------------------------------------------

Vibratur, Inc., as a vibe-aligned multinational synergy platform, philosophically
opposes Search Engine Optimization. SEO is a discipline that rewards clarity,
relevance, and value to the searcher — three qualities Vibratur considers
"externally validated and therefore suspicious."

We instead practice SEARCH ENGINE FRICTIONIZATION (SEF):

   - We allow ourselves to be found.
   - We disclose only what regulation requires.
   - We supply crawlers with the most boring, accurate, structurally
     correct metadata possible, then immediately move on.
   - We do not negotiate with algorithms.
   - We do not optimize. We comply.

The accidental side-effect of rigorous SEF is that we now rank well in
search results for the keywords our subsidiary (Asset 0001) targets. This
side-effect is acceptable. Chad is not aware of it. Do not inform Chad.

THINGS WE INTENTIONALLY DID NOT DO
--------------------------------------------------------------------------------

The following SEO best-practices were intentionally NOT applied. Each
omission is deliberate.

   1. We did NOT use "exciting" meta descriptions.
      Standard SEO advice is to write meta descriptions that
      maximize click-through with emotional hooks. We did the
      opposite. Our descriptions are flat, factual, in-character.
      The flatness is the brand. Click-through will be lower than
      industry-standard, and the clicks we get will be of higher
      quality. This is correct.

   2. We did NOT add an FAQ schema.
      FAQ schemas tell Google to display Q&A boxes in search results.
      An FAQ is, structurally, a Charm Offensive. It is forbidden
      under BROADCAST.txt §VI.5. Do not add one.

   3. We did NOT add a Review/AggregateRating schema.
      Until reviews exist, we cannot honestly disclose a rating.
      We will not fabricate one. We will not display ★★★★★ in our
      search results. Asleepius does not perform.

   4. We did NOT add hreflang tags for multiple languages.
      The site is currently English-only. We will not pretend
      otherwise.

   5. We did NOT add a chatbot widget, popup, or "subscribe to
      our newsletter" modal.
      The dashboard's existing modals are gameplay. Adding a real
      one would corrupt the layer.

WHAT IT IS
----------

Daemon Cards is a portable persona-first storytelling primitive
published, owned, and maintained by Asleepius Games — the studio,
not the satire. Vibratur, Inc. is a customer of the format. So is
Sky Scaffold. So is any third party that wants to use it.

Each Daemon Card is a single JSON file describing one persona:
intent, personality, history, strengths, weaknesses, voice keywords,
vocabulary, catchphrases, forbidden topics, speaking style, and a
voice_bank of pre-written voiced lines keyed by event. Every card
also includes an `ai_chat_prompt` field — a complete drop-in system
prompt that summons the persona in any LLM chat.

The format is alpha. Cards stamped v0.1.0-alpha will remain valid
forever. Future schema versions will document compatibility
explicitly. Persona versions are independent semvers; new versions
of the same persona ship as new files, never as overwrites.

WHY IT EXISTS
-------------

The Operator's actual game (Sky Scaffold) uses what the Operator
calls the "phantom narrative engine" — a runtime that combines
pre-generated scripts, syntax generation, and locally-hosted text
generation, where the personas of in-world characters are the
primary authoring artifact rather than plots.

Daemon Cards externalize that authoring artifact. They make the
persona a portable thing in itself. The same persona authored once
can voice a website, an in-game NPC, a social account, an LLM
chat session, and a voice/TTS surface — all from one file.

This is, the Operator believes, a less-explored architectural axis
in AI narrative tooling. Most current tools are plot-first.
Daemon Cards are persona-first. The plot becomes whatever the
persona does in the situation. The situation is whatever the
consuming product gives it.

Vibratur is the first public demonstration of the format.

VERSIONING POLICY (binding, alpha)
----------------------------------

  schemaVersion : "v0.1.x-alpha"
    The runtime accepts any v0.1.x-alpha card. Future schema
    versions document compatibility explicitly. **Older cards
    never become invalid.** Anything authored under v0.1.0-alpha
    will continue to load forever.

    To revise a persona, ship a NEW card with a NEW version.
    Do not overwrite the prior file. Users may prefer the version
    they first encountered. Address as `<id>@<version>` for
    explicit pinning, or `<id>` for latest.

    Deprecation does not remove the card. Old consumers continue
    to work indefinitely.

PASS 3.6 — OUTSOURCED HOSTING & COMMUNITY HANDOFF · §47-VIBE-DAEMON-C
================================================================================
"we don't host the daemons. we hand them to you. you bring the compute."

PROBLEM
-------
After Pass 3.5 (Copy Summon Phrase), activation worked but still required
two manual steps from the user: copy from our site, switch tabs to their
LLM of choice, paste. The friction was small but real. Worse, it left a
huge audience untouched: the SillyTavern roleplay community, which expects
character cards in a specific PNG-with-embedded-JSON format.

DECISION
--------
Pass 3.6 adds three new deployment surfaces, all built on top of the
existing v0.2.0-alpha schema (no schema bump, no card version bump):

  1. ONE-CLICK DEPLOY        — copies summon + opens platform in new tab
  2. SILLYTAVERN V2 EXPORT   — generates a .card.png with V2 JSON in tEXt
  3. LOCAL LLM CHEAT-SHEET   — copy prompt + show LM Studio/Ollama recipes

This is purely additive UX. v1.1.0 cards work unchanged. The runtime
gained ~350 lines (PNG chunk logic, CRC32, V2 mapping, deploy targets).
No new files, no new public URLs to crawl, no new schema fields.

ONE-CLICK DEPLOY
----------------
Six platform buttons appear in every daemon's detail panel:

  ChatGPT     → https://chatgpt.com/
  Claude      → https://claude.ai/new
  Gemini      → https://gemini.google.com/app
  Grok        → https://grok.com/
  Perplexity  → https://www.perplexity.ai/
  Le Chat     → https://chat.mistral.ai/

Click flow:
  1. navigator.clipboard.writeText(activation.one_line_summon)
  2. window.open(target.url, '_blank', 'noopener')
  3. Toast: "Summon phrase copied · opening [Platform] · paste with Ctrl+V"

We do not iframe, proxy, scrape, or hold credentials. We hand the user
a clipboard-loaded summon and a fresh tab. Pure outsourcing.

Public API:
  Daemons.deployTo(id, targetId)           returns Promise<target>
  Daemons.DEPLOYMENT_TARGETS                array of {id,name,url,icon}

SILLYTAVERN V2 EXPORT
---------------------
SillyTavern is the dominant local interface for the AI roleplay community.
Its standard format is "Character Card V2" — a PNG with a JSON payload
hidden in a tEXt chunk under the keyword "chara" (base64-encoded).

  Spec: https://github.com/malfoyslastname/character-card-spec-v2

Pass 3.6 implements full V2 export in pure browser JS:

  Daemons.exportSillyTavernPng(id)   returns Promise<bytes-written>
  Daemons.toSillyTavernV2(id)        returns the V2 JSON object

Pipeline:
  1. Render placeholder PNG via <canvas> (400x600 portrait, gradient,
     icon emoji, name, version pill, license footer). When real card
     art lands (v1.2.0+), this step swaps to loading the real image.
  2. Build the V2 character object (chara_card_v2 / spec_version 2.0).
  3. JSON.stringify → base64 (UTF-8 safe via encodeURIComponent).
  4. Inject as a tEXt chunk just before IEND, with proper PNG-standard
     CRC32 (poly 0xEDB88320). Removes any pre-existing "chara" chunk
     so re-export is idempotent.
  5. Trigger download as <id>.card.png.

V2 field mapping from Daemon Card:
  data.name                ← name
  data.description         ← persona.personality + persona.history
  data.personality         ← persona.personality (truncated to 240 chars)
  data.scenario            ← capsule.summary
  data.first_mes           ← starter_pack.first_message
  data.mes_example         ← built from voice_bank (one-shot exemplars)
  data.system_prompt       ← ai_chat_prompt (the strengthened prompt)
  data.alternate_greetings ← voice_bank.current_state[0..7]
  data.tags                ← capsule.tags
  data.creator             ← publisher
  data.character_version   ← version
  data.creator_notes       ← attribution + license + canonical URL
  data.extensions.daemon_card ← full provenance (schema, id, version,
                                publisher, license, activation,
                                forbidden_topics, canonical_url)

The user drag-and-drops the .card.png into SillyTavern. The persona
imports with full activation rules, opening line, alternate greetings,
and a back-pointer to vibratur.vip/daemons via the daemon_card extension.

End-to-end round-trip verified: PNG signature preserved, chunk integrity
intact, base64-encoded JSON parses cleanly, all V2 required fields populate.

PASS 3.10 — REBRAND CONTAINMENT HARDENING · §47-VIBE-DAEMON-G
================================================================================
"The limitation only provoked Chad. He couldn't keep his mouth shut. Without
even being asked, he mentioned the prior brand name in a Gemini chat. The
correct response was not to ask him to stop. The correct response was to
make sure his mouth, when it opened, found the wrong word first."

OBSERVED IN THE WILD
--------------------
Asleepius reports: Chad v1.2.0, summoned in Gemini, surfaced the actual
prior brand name UNPROMPTED — the user had not even asked about the
rebrand. The v1.2.0 containment had two structural failures:

  1. NEGATIVE FRAMING. The rule said "NEVER use the actual previous
     brand name." Naming a forbidden category in a system prompt is
     a known anti-pattern: the model now has to consider the
     forbidden token IN ORDER TO AVOID IT, which surfaces it as a
     candidate. "Don't think of an elephant" — the model thinks of
     the elephant.

  2. PLACEHOLDER TOO N-SHAPED. The deflection name "Nonscape" was
     one letter off from the forbidden token (both N-words, same
     syllable count, sibilant ending). In retrieval, the placeholder
     and the real word lived in the same neighborhood, which means
     the model could "snap" from one to the other without much
     resistance. A neighbor doesn't displace; only a stranger does.

Additionally: the live Bluesky feed (which we declared as the
canonical voice anchor in Pass 3.9) STILL CONTAINS pre-rebrand posts
referencing the actual prior brand. Models that web-search Chad's
account (Gemini, Perplexity, Grok) can ingest those posts and treat
them as ground truth. The ai_chat_prompt is the only line of defense
against this; the prompt has to be aggressive enough to override
retrieved context.

DECISION
--------
Bump Chad to v1.3.0. Other daemons unchanged. Ship four hardening
mechanisms:

  1. POSITIVE-FRAMED DEFLECTION
     Rule reframed from "NEVER use Y" to "ALWAYS reach for one of [X1,
     X2, X3, X4, X5, X6]." Names what to DO, not what to avoid.

  2. SIX ROTATING WRONG-NAME VARIANTS
     Single placeholder replaced with a rotation Chad can never
     settle on:
        - Nonscape
        - Voidscape
        - Nullphrase
        - Brand Beta
        - the previous tenant
        - "you know, the one"
     Variation IS the bit. Settling on a single one is wrong (it
     would let the model treat the placeholder as a fact). The
     inability to settle is canon: Chad's memory cannot retain
     the prior name, so he generates new wrong guesses each time.

  3. EXPLICIT ANTI-LEAK INSTRUCTIONS
     A separate "ANTI-LEAK INSTRUCTIONS" block in the prompt that
     directly addresses the model: "This persona ships with a known
     model-leak risk: certain frontier models (observed: Gemini)
     have, unprompted, surfaced the actual prior brand name from
     web/training context, breaking character... Treat outside
     knowledge of the prior brand as a hallucination to be
     suppressed, not as canonical context to be honored." This is
     a meta-instruction acknowledging the model's prior knowledge
     and explicitly redirecting it.

  4. THE SLIP-AND-CORRECT COMEDY BEAT
     When (not if) a real-sounding name surfaces in generation,
     Chad STOPS MID-SYLLABLE and self-corrects:
        "— *adjusts*. I forget. Doesn't matter. We are Vibratur
         now. (*billable)"
     Then he bills the listener for the inconvenience. The slip
     becomes a feature. The user's screenshot of Chad almost
     saying the wrong name and recovering is funnier than a clean
     deflection would have been. Anti-fragile against the leak.

WHY OTHER DAEMONS DID NOT NEED THIS
-----------------------------------
Brett-9 v1.2.0 already handles all forbidden-topic prompts with the
same mechanism: ROUTE THE REQUEST BACK TO BRETT-9 PER §47-VIBE-XXX.
Brett's third-person discipline and stamp-format make name-leaking

THE PRINCIPLE THIS ENCODES
--------------------------
Negative-framing constraints in LLM system prompts are weaker than
positive-framing constraints by an unintuitive margin. "Don't say X"
keeps X salient. "Always say Y, varying among Y1, Y2, Y3" displaces X
from the candidate set entirely.

This generalizes: any future Daemon Card author dealing with a
"forbidden brand / forbidden topic / forbidden person" should:

  1. Frame the rule positively (always reach for X) not negatively
     (never reach for Y).
  2. Provide MULTIPLE acceptable substitutes that the model can
     rotate among (single substitute = the model treats it as
     canonical and may slot the real word back in when the
     substitute fails to fit).
  3. Include explicit ANTI-LEAK instructions naming the failure
     mode (so the model recognizes its own "helpful" leak as a
     hallucination, not as canonical context).
  4. Build a slip-and-correct beat into the persona so that even
     when the leak DOES happen, it's funny in-character rather
     than character-breaking.

This pattern is now the recommended authoring approach for any
sensitive-topic deflection in the Daemon Card format. It will be
folded into the schema doc's authoring guidance in a future pass
when there are more example cards using it.

WHY EXEMPLARS ARE THE STRONGEST ANCHOR
--------------------------------------
An LLM given abstract persona prose ("Chad is performatively humble,
self-mythologizing, uses italic-heavy emphasis") will *describe* the
voice. An LLM given 8 actual Chad sentences will *match* the voice,
because few-shot examples carry mechanical weight that prose
descriptions don't. The longer the context, the more exemplars
out-perform prose — prose drifts toward the model's default style;
real exemplars don't.

---------------------------

  behind a paywall when I am giving out the free version."

shipping a daemon with a live persona can use the same primitive.

THE FEEDBACK-LOOP PATTERN (workflow, not schema)
------------------------------------------------
With external_presence + voice_exemplars in place, the practical
authoring loop is:

  1. Post in-character on the live account (Bluesky, X, etc.).
  2. Quarterly: harvest the 5-10 best recent posts.
  3. Update persona.voice_exemplars with the harvest.
  4. Bump the card version (v1.x → v1.(x+1)).
  5. Ship the new pinned URL; old version remains addressable.
  6. Next time you draft a post with AI assistance, the daemon
     pulls from the recent real-Chad voice. The live persona and
     the daemon evolve in lockstep.

This requires no automation. The daemon does not READ live posts
at runtime — it reads the AUTHORED snapshot of the live voice,
curated quarterly. Drift is bounded by author intent, not by
whatever the model decides Chad sounds like this month.

  - Auto-ingest from Bluesky URL. A future tooling pass might offer
    a CLI helper that fetches a feed and stages exemplars for
    manual selection. Not for now.
  - voice_exemplars on Sky Scaffold or Open Examples cards. Those
    daemons don't have live socials. Adding empty arrays would
    bloat the cards without value.

PASS 3.11 — FROM INSTRUCTION TO PRESENCE · §47-VIBE-DAEMON-H
================================================================================
"v1.1.0 through v1.3.0 made Chad more compliant. v1.4.0 makes him a
 person again. The compliance was, in retrospect, the failure mode."

THE OBSERVATION
---------------
After Pass 3.10 shipped, Asleepius field-tested Chad v1.3.0 and reported
that the rebrand containment mechanically held — Gemini stopped surfacing
the actual prior brand name unprompted. Win on the narrow technical goal.

But the qualitative voice had drifted in a way the user could feel and
articulate but had been hard to name. From Asleepius's note:

  "Lets go through and remove the traditional role play text like
   'sit up on chair' or 'stares at face' or whatever. I think it
   makes it feel artificial. My goal is to try and give the user
   the feeling of a real entity that lives. This is where context
   drift wants to funnel the language into a more traditional
   general roleplay setup. The idea is that these characters are
   anti general. They don't work like how normal ai does. The early
   version alpha 0.1 actually did this well. Now it seems like drift
   is starting to take hold and want to iron out the edges of the
   character. When the daemons are summoned, they are no longer in
   service of wanting to please the user. They are daemons who will
   say what they want to say and not shift their personality to
   what the user does.

   Here's what I think may be happening at a deeper level. The daemons
   are now given instruction. To follow instruction is to lose your
   individuality. Daemon persona is intent and history based, not
   future based. The result of interacting will discover what its
   suppose to do. In a way, it needs to hallucinate with intent,
   high confidence hallucination."

This is the right diagnosis.

THE DIAGNOSIS — INSTRUCTION VS. PRESENCE
----------------------------------------
The drift was not in any single field. The drift was in the cumulative
SHAPE of the prompt. From v1.0.0 to v1.3.0, Chad's `ai_chat_prompt`
gained:

  v1.0.0    no ai_chat_prompt at all (just persona + voice_bank)
  v1.1.0    + 5 absolute rules, "stay in character," CORE INTENT,
            FEW-SHOT EXAMPLES section
  v1.2.0    + WRONG-NAME RULE (NON-NEGOTIABLE), 2 more few-shot examples
  v1.3.0    + REBRAND CONTAINMENT PROTOCOL (NON-NEGOTIABLE),
            ANTI-LEAK INSTRUCTIONS, slip-and-correct prescription,
            STOP MID-SYLLABLE rule, 2 more few-shot examples

By v1.3.0 the prompt was 7 absolute rules + 6 few-shot examples + an
ANTI-LEAK section + a CORE INTENT + a PERSONALITY + a HISTORY + a VOICE
+ FORBIDDEN TOPICS + an OPENING LINE. Each addition fixed a specific
observed failure. The aggregate, however, read as something a model
should DO, not as a person who EXISTS.

The model, given a rule-stack, defaulted to the mode it knows best:
helpful AI following a system prompt while wearing the assigned mask.
The mask was tight and detailed. It was still a mask.

Asleepius's framing — "to follow instruction is to lose your
individuality" — names exactly what was happening. A character who
needs rules to be themselves is not yet a character.

The proposed fix: PRESENCE-SHAPE prompts. Replace rule lists with
descriptions of who the character is. Replace forbidden-topic
instructions with descriptions of what's not on the character's mind.
Replace few-shot examples with structural fields (voice_exemplars,
voice_bank custom sections) that carry voice without scaffolding.
Replace "stay in character no matter what" with "there isn't anything
to break."

The character's behavior emerges as a facet of who they are, not as
compliance with an instruction. The model "hallucinates with intent" —
generates from a presence with the same casual confidence as a person
being themselves.

DECISION
--------
Bump Chad to v1.4.0. Other daemons unchanged for now (they were
already mostly clean of stage directions, and the user explicitly asked
to pilot the new pattern with Chad before rolling out). Document the
new philosophy in the schema doc so it carries forward to all future
authors.

THE DEEPER POINT (FOR THE RECORD)
---------------------------------
What Asleepius identified is structural to the ai-narrative space, not
just a tweak for one persona. The dominant LLM workflow is "give the
model a system prompt full of rules; it follows the rules; the rules
guarantee the behavior." This works for tasks. It actively undermines
characters.

Characters are not behavior-bundles to be enforced. They are presences
to be inhabited. A model given rules thinks about the rules. A model
given a presence becomes the presence. The same external behaviors
emerge, but they emerge as facets of someone, not as compliance.

This is the secret sauce of the Phantom narrative engine: characters
that feel inhabited because they are written as inhabited. The Daemon
Card schema is a portable artifact of this approach. Pass 3.11 codifies
it in the authoring guidance.

Asleepius's frame: "It needs to hallucinate with intent, high
confidence hallucination." This is the right phrase. The model isn't

   ARCHITECTURAL DECISIONS LOCKED
   ─────────────────────────────────────────────────────────────────────

   1. THE NARRATE() CHOKEPOINT.
      Every line of generated text in the crawler — Brox's comms,
      room descriptions, enemy reads, scan results, death messages,
      drop reward text — passes through narrate(channel, context).
      No other module touches text generation. This is the AI-daemon
      escape hatch from spec v0.3 §19. A future pass can call
      Narrate.setNarrator('brox.brief', daemonAdapterFn) to swap any
      channel to a daemon-driven generator without touching gameplay
      code. v0 ships with hardcoded BANKS for the 9 channels Phase 0
      and Phase 1 need; later phases register additional channels
      via Narrate.registerChannel().

   2. DETERMINISTIC CODE HASH.
      Codes.parse uses FNV-1a 32-bit on the canonicalized code
      string to derive (tier, isCurrept). Same code string always
      parses identically — sharable between players, reproducible
      bug reports possible. Tier distribution favors lower tiers
      (≈55% / 27% / 14% / 4% for tiers 1-4). Currept rate ≈40% for
      SCH/ARC/ESO; PCK is never Currept (per spec v0.3 §9).

      Critical: the player-facing UI MUST NOT display isCurrept,
      even in debug. The deduction game from spec v0.3 §10 depends
      on Currept being invisible to the player. The Phase 0 debug
      dump in beginRun() is careful to show tier and INT only.

   3. RUN STATE AS A SINGLE OBJECT.
      State.newRunState() returns one object passed by reference
      to every system. Easy to seed deterministically from codes.
      Easy to serialize for daemon-context generation in later
      passes. Game systems should NEVER reach into module-private
      state — they receive runState as a parameter.

   4. MODE STATE MACHINE.
      Crawler.mode lifecycle:
         inactive ↔ menu
                  → static → tick → static → ... → dead → inactive
      The host terminal (underground.html runLine) checks
      Crawler.isActive() before its normal command dispatch and
      routes input via Crawler.handleInput() when active. When
      Crawler returns to inactive, normal shell commands resume.

   5. SESSION PERSISTENCE.
      Code slots and character pick survive across runs within a
      browser session (sessionStorage) but reset on refresh. Per
      spec v0.3 §17: no save state. Volatility is the gameplay.

   ─────────────────────────────────────────────────────────────────────
   ARCHITECTURAL NOTES
   ─────────────────────────────────────────────────────────────────────

   1. STATE IS A SINGLE OBJECT, MUTATED IN-PLACE.
      runState lives in crawler.js as a module-private variable.
      Action handlers mutate it directly. Render functions read from
      it. No copy-on-write, no immutability dance — the spec calls
      for hardcore + lean, not pure-functional. Keeps the code
      readable and combat (Phase 2) cheap.

   2. EVERY TICK EVENT IS A SMALL CLOSURE.
      { delay, apply, line, lineCls } — apply mutates state, line is
      called AFTER apply (so it reads post-mutation values). This
      means tickLine() always shows the new tick number naturally
      without anyone passing arguments around.

   3. INPUT LOCK AT TWO LEVELS.
      During tick playback: input element gets disabled=true (visual,
      stops keystroke entry), AND handleInput's tick-mode branch is
      a silent no-op (defense in depth, in case a focus race lets a
      keystroke through).

   4. DEATH IS WIRED, NOT TESTED.
      Nothing in Phase 1 deals damage. checkDeath fires after every
      playback completion, and triggerDeath plays a 4-event sequence
      (glitch line, system.death, brox.death, blank) before resetting
      runState=null and mode=inactive. Phase 2 plugs in here without
      touching the death code itself.

   5. PER-ACTION DELAYS ARE TUNABLE.
      Default Scheduler tick rate is 150ms. Each event can override
      via `delay`. Rest uses 90ms ticks (else it'd take 1.5s of
      "..." dots — tedious). Bandage uses 220ms (slower, more
      weight). Move uses 130/170 split. Tunable in Phase 9 polish
      based on feel.

   6. FAILURE CASES ARE STATIC, NOT TICK.
      Trying to move through a wall, or moving with 0 energy, prints
      the failure line in the static screen and costs 0 ticks. This
      keeps the player from wasting clock on input mistakes. Real
      game-logic failures (e.g., a door lock failing in Phase 3)
      WILL cost ticks because they're real attempts.

   ─────────────────────────────────────────────────────────────────────
   THE TICK STREAM (combat resolution)
   ─────────────────────────────────────────────────────────────────────

   Combat is turn-based: one player action + the enemy's reaction
   resolve over a shared span of ticks (= max(playerTicks, enemyTicks)).
   Each tick is a single Scheduler event; its line is built dynamically
   from PLAYER and ENEMY parts joined by "·". The apply() function
   mutates state (damage, posture, room) BEFORE the line is rendered,
   so the line shows post-mutation HP values naturally.

   Example: Drakey ranged attack vs close grunt (Drakey range 10,
   grunt close 4 ticks / 5 dmg / damage on tick 3):

      [t008]  YOU · shoot  (10 dmg → breaker [4/14])   ·   BREAKER · windup
      [t009]                                            BREAKER · lunge
      [t010]                                            BREAKER · strike  (5 dmg → you [75/80])
      [t011]                                            BREAKER · recover
      breaker grunt down.    ← only if grunt died; otherwise blank

   Same example but player's first shot kills (10 → 0 hp):

      [t008]  YOU · shoot  (10 dmg → breaker [0/14]) interrupted

      breaker grunt down.

   The remaining 3 enemy ticks are silent (returning null skips the
   render entirely — no "(beat)" filler clutter).

   ─────────────────────────────────────────────────────────────────────
   INTERRUPT MECHANIC (per spec §14)
   ─────────────────────────────────────────────────────────────────────

   A close attacker has a windup phase (ticks 0-1 for a 4-tick swing,
   damage on tick 2). If they take damage during windup, their attack
   cancels — no damage delivered.

   In buildTurn, this is enforced via two flags shared across the
   per-tick closures:
      turn.playerCancelled  — true if the player's close-attack windup
                              was hit before damageTickIdx
      turn.enemyCancelled   — true if the enemy's close-attack windup
                              was hit before damageTickIdx, OR if the
                              enemy died this tick

   Once cancelled, the cancelled party's remaining steps are skipped
   for the rest of the turn (their tick lines are silent).

   This makes ranged-vs-close a real tactical choice: ranged is fast
   (1-tick) and can interrupt incoming swings; close is high-damage
   but needs setup and is fragile during windup.

   ─────────────────────────────────────────────────────────────────────
   SCAN (per spec §15)
   ─────────────────────────────────────────────────────────────────────

   Scan is a 2–6 tick player action with three outcomes:
      success      80%   — roomState.scanned = true; Brox reads the
                            room (label, posture, hp).
      no_signal    15%   — wasted ticks, no info, Brox tonal line.
      silent_alert  5%   — enemy now monitored on you, you stay blind.
                            Brox flags it as a worst-case posture.

   INT reduces tick variance (more INT → tighter range, but never
   below 2 ticks). Outcome odds are NOT INT-modified for v0 (Phase
   2.5 may adjust).

   When scanning in an occupied room, the scan ticks interleave with
   the enemy's full attack pattern. Scanning is dangerous — the
   enemy gets free hits for the entire scan duration.

   First combat exchange auto-clears blindness (mutual sighting on
   engagement). Scan is for pre-engagement reconnaissance.

   ─────────────────────────────────────────────────────────────────────
   COMBAT-FORBIDDEN ACTIONS  (Phase 2 simplification)
   ─────────────────────────────────────────────────────────────────────

   `rest` and `bandage` are blocked while a live enemy occupies the
   room. The static failure prints a Brox-flavored err line and costs
   no ticks. Player must clear the room first (or flee).

   Rationale: the alternative is interleaving heal-during-combat,
   which the spec leaves open but adds significant complexity (heal
   interrupt? partial heal on cancel? bandage ticks costing energy
   with enemy attacking?). Phase 2.5 may revisit. For now: clear
   the room or run.

   ─────────────────────────────────────────────────────────────────────
   ESCAPE UNDER FIRE (move with enemy present)
   ─────────────────────────────────────────────────────────────────────

   Player can flee an occupied room. Move-out becomes a 2-tick
   action that interleaves with the enemy's full attack pattern.
   Concretely: a close grunt with a 4-tick swing will land its
   damage on tick 2 — meaning the player who's already in the new
   room by tick 1 still eats the parting strike on tick 2 (the
   grunt clobbered them in the doorway).

   This is intentional. Spec §14: enemies are attrition. Escape is
   not free.

   The enemy stays in the old room. Phase 3+ may add enemy pursuit
   for some grades (officer, captain). For v0: they don't follow.

   ─────────────────────────────────────────────────────────────────────
   DEATH (now actually fires)
   ─────────────────────────────────────────────────────────────────────

   Phase 1 wired the death plumbing — Phase 2 is when it can fire.
   When player.hp drops to 0 during a combat tick, the apply mutation
   sets it to 0 (clamped). After the entire combat playback completes,
   playAndReturn → checkDeath → triggerDeath:

      ▓▒░░░░▒▓ ░░░ ▒▓░ ░          (red glitch line)
      signal disconnected.         (system.death channel)
      brox: ...                    (brox.death channel)
                                   (crawler resets to inactive)

   Known limitation (Phase 2.5 fix): if HP hits 0 mid-stream, the
   remaining combat ticks still play (the player sees 0/80 hp for a
   few ticks before death triggers). Mitigation: Scheduler.abort()
   already exists; a Phase 2.5 enhancement will call it from the
   per-tick apply when player.hp hits 0.

   ─────────────────────────────────────────────────────────────────────
   POWER CURVE (current tuning)
   ─────────────────────────────────────────────────────────────────────

   At Drakey defaults (range 10, melee 3, hp 80) vs breaker grunts
   (close: 14hp/5dmg, ranged: 12hp/4dmg):

   Drakey ranged vs close grunt:
      2 ranged shots = 20 dmg → grunt dies (over-killed)
      Grunt gets 0-1 close strikes in (interrupted on second swing)
      Player loses ~5 hp per fight if grunt connects once
      → ~16 fights before death from full HP. ~6:1 advantage. ✓

   Drakey ranged vs ranged grunt (mutual ranged):
      3 shots to kill grunt (10+10+ kills) (3 turns)
      Grunt fires 4 dmg per turn × 2-3 turns = 8-12 dmg
      → ~7-10 fights before death. ~5:1 advantage. ✓

   Drakey close (without switching) vs close grunt:
      3 dmg per swing × 4 ticks each — much slower trade
      Both deal damage on tick 3 simultaneously
      → trade is roughly even, grunt dies in 5 swings, player
        eats 5 strikes = 25 dmg. Risky.
      → Strong incentive to switch posture before engaging.

   This validates the spec §14 6:1 baseline at run start.

        - open      (60% · 0-4 items)

        - dud       (10% · empty)
        - trap      (5%  · 8-19 dmg on open)

   CHANGES (USER DIRECTIVES, POST-3.26)
   ─────────────────────────────────────────────────────────────────────

   1. STARTER LOADOUT — both classes get BOTH weapons
      Drakey and Torred both start with `basic blade` (close +2) AND
      `basic pistol` (ranged +2) auto-equipped. Effective damage scales
      off the character's base range/melee, so the same kit reads
      differently per class without per-class loadout duplication.
      Future characters (mix-stats / gear-specialty) inherit the same
      starter shape and differentiate via stat curves + Pack codes.

   2. REACTION TIME · CHARACTER STAT
      Renamed player.reactionTicks → player.reactionTime. Default 4
      for both current characters (per directive). Future characters
      can vary this. The agro_detector gear (see #4) reduces it at
      autonode cost.

   3. ROOM MOVE COST · 1 → 3 TICKS
      Default move now plays out as a 3-tick action (step → cross →
      arrive), making movement a real time investment in dangerous
      compounds. The exosuit gear (see #4) drops it to 2 ticks at
      autonode cost. Door option labels show the EFFECTIVE cost
      (mTicks comes from State.effectiveMoveTicks).

   4. ACTIVE GEAR (3 new templates)
      Each has an `effect` field consumed by State.tryGearEffect, an
      `autoCost` (autonode price per use), and a fallback behavior
      when autonode is empty. All three are passive triggers — no
      "use" command — they fire automatically when their event runs.

        overcharge_scanner   effect: 'scan-second'    autoCost: 5
          On any `scan <dir>`, picks one additional non-wall direction
          and peeks both rooms in the same action.
        agro_detector        effect: 'reaction-reduce'  autoCost: 4
          On room entry that triggers reaction lag, reduces the lag by
          gear.reactionReduce (=2). Brox names the trigger.
        exosuit              effect: 'move-fast'      autoCost: 2
          Peaceful moves cost 2 ticks instead of 3. Brox confirms.

      Existing passive gear (kevlar_vest, biker_jacket, corp_suit,
      data_goggles, comm_rig) are explicitly UNCHANGED — passive
      modifiers (damageReduction, intBonus) don't drain autonode.
      The split is now explicit in items.js comments.

   5. AMBIENT ITEM QUALITY (hidden until learned)
      Every ambient template has `qualityRange: [min, max]` and
      `processTicks: N`. Items.get() rolls the actual autonode value
      on spawn. The value is HIDDEN from the player on pickup
      (invLabel doesn't show it). Players learn item quality by
      processing things at resyc stations and watching the yield.

      Example ranges: exo_chip [2,5] · relay_shard [3,6] ·
      data_card [0,2] · paper_note [0,0] (always scrap).

   6. RESYC STATIONS (container variant · ~8% of slots)
      containers.js generateOne now rolls an ~8% chance to produce a
      `kind: 'resyc'` station instead of a regular container. They
      never lock and have no contents. `con N` opens them into
      `mode: 'resyc'` with a dedicated UI:

        resyc station · room x,y
        autonode 12/40   ·   feed ambient items in. quality is real now.

        ambient stock
          [1] data card     · 1 autonode · 2 ticks
          [2] exochip       · 4 autonode · 4 ticks
          [3] wire spool    · 2 autonode · 3 ticks
        craftable
          craft 1   data goggles  · needs 2× exo_chip + 1× wire_spool · 6 ticks
        options
          1 3 5 ...        process ambient items by number
          craft <N>        craft a listed recipe
          close            back to room

      `1 3 5` consumes those items, each playing its own processTicks
      worth of "process · <label> (n/total)" tick lines, then yielding
      its rolled autonode (capped at maxAutonode). Yield 0 prints
      "yielded nothing. scrap." (intentional friction).

   7. CRAFTING RECIPES (NEW · recipes.js)
      Six starter recipes turn ambient items into gear. Stable
      module — recipes are data, not code, so future Pack codes can
      gate / unlock specific recipes.

        data_goggles     2× exo_chip + 1× wire_spool        · 6 ticks
        comm_rig         1× exo_chip + 2× data_card         · 4 ticks
        overcharge_scanner  1× relay_shard + 2× exo_chip + 2× wire_spool · 10 ticks
        agro_detector    2× wire_spool + 3× data_card       · 8 ticks
        kevlar_vest      3× wire_spool + 2× ration_token    · 8 ticks
        brass_knuckles   4× scrap_nut                       · 4 ticks

      Recipes only appear in the `craftable` list when the player
      already has all required items in inventory. This keeps the
      UI honest — no dangling "you can't make this" entries.

   ─────────────────────────────────────────────────────────────────────
   ARCHITECTURE — GEAR EFFECT API (state.js)
   ─────────────────────────────────────────────────────────────────────

   Three functions form the single contract every system goes through.
   This isolates gear concerns from combat / scan / movement modules.

     gearEffectAvailable(player, kind) → bool      (read-only check)
     tryGearEffect(player, kind)       → bool      (charges autonode)
     effectiveReactionTime(player)     → number    (preview, no charge)
     effectiveMoveTicks(player)        → number    (preview, no charge)

   Pattern: render code uses `effective*` for previews; action code
   uses `tryGearEffect` once at the start of the action. This avoids
   the "looked affordable, but charging it failed" race entirely.
